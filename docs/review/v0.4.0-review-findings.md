## BLOCKERS (must fix before ship — auto-fix scope)

### B1 — `_call_with_timeout` doesn't enforce the timeout ✅
**`summarize/llm_summary.py:202-211`** · violates invariant *"LLM calls never hang the UI."*

```python
with concurrent.futures.ThreadPoolExecutor(max_workers=1) as pool:
    future = pool.submit(func)
    return future.result(timeout=timeout_seconds)   # raises TimeoutError at 30s...
# ...but `with ... as pool` __exit__ calls shutdown(wait=True), which BLOCKS
# until the worker thread (the actual LLM HTTP call) finishes anyway.
```

Both the correctness and security reviewers reproduced this independently: a 2–3s worker with a 0.5s timeout still takes the full 2–3s to return. Combined with `LLM_MAX_RETRIES=3`, an HTTP handler thread can block for **90–360s** on a stalled network — defeating the entire timeout/retry feature this sprint added.

**Fix:** drop the context-manager form; on timeout call `pool.shutdown(wait=False, cancel_futures=True)` (Py 3.9+) so the caller is actually unblocked:
```python
pool = concurrent.futures.ThreadPoolExecutor(max_workers=1)
future = pool.submit(func)
try:
    return future.result(timeout=timeout_seconds)
except concurrent.futures.TimeoutError as exc:
    future.cancel()
    pool.shutdown(wait=False, cancel_futures=True)
    raise TimeoutError(f"LLM call timed out after {timeout_seconds}s") from exc
```
> Caveat: a thread blocked in a C-level socket read can't be force-killed; the real bound is the SDK's own timeout (already set). This fix unblocks the *caller* — the belt-and-suspenders backstop actually works.

### B2 — Inline 404 fallbacks return 500 instead of 404 ✅
**`app/server.py:309,311,358,378,394,400,402`**

```python
self._respond({"ok": False, "error": "Not found"}, 404)   # 2 positional args
```
…but `def _respond(self, result):` takes **one** arg (a dict, or a `(dict, status)` tuple). The two-arg call raises `TypeError`, which the verb method's `except Exception` swallows into a generic **500**. Every unmatched-subpath 404 in `do_GET/do_POST/do_PUT/do_DELETE` is actually a 500.

**Why nothing caught it (root cause → TG2 + TG3):**
- mypy passed clean because `def do_GET(self):` is **unannotated**, and mypy (non-strict) **skips the bodies of unannotated functions** — so the bad `_respond` calls were never analyzed.
- T3 asserts on `handler._get_session_detail("missing-id")` returning `(payload, 404)` *directly*, never routing through `do_GET` → `_respond`. The handler returns the right tuple; the dispatch that sends it is broken.

**Fix:** wrap the literal in a tuple at all 7 sites: `self._respond(({"ok": False, "error": "Not found"}, 404))`.
(Handler-returned 404s like `self._respond(self._get_session_detail(id))` are fine — they pass one tuple arg.)

### B3 — `session.py` has its own unguarded LLM calls ✅
**`app/session.py:614-637`** · violates invariant *"LLM calls never hang the UI"* + *"mlx-lm serialized."*

The sprint hardened `llm_summary.py`, but `session.py` carries **duplicate** `_call_gemini` / `_call_claude` with **no timeout, no retry, no error mapping**:
```python
def _call_claude(prompt, model_id):
    client = Anthropic(api_key=api_key)
    message = client.messages.create(model=model_id, max_tokens=4000, messages=[...])  # blocks forever
    return message.content[0].text
```
Reached via `_generate_custom_notes` (`session.py:597-602`) → the `POST /api/sessions/<id>/generate-notes` and notes endpoints, **on the HTTP request thread**. A stalled provider hangs that thread indefinitely — exactly the failure mode B1/the resilience work was meant to prevent.

**Fix:** delete `session.py`'s `_call_gemini`/`_call_claude` and route through the guarded `llm_summary._call_llm_gemini` / `_call_llm_claude` (timeout + retry + error mapping). Also confirm `session.py`'s local path uses the semaphore-guarded `_call_llm_local`.

## WARNINGS (real, non-blocking — report-only this round)

- **W1 — Range parser** (`server.py:484-503`): `int(parts[1])` raises on multi-range / missing `-` → 500; suffix form `bytes=-500` mishandled as `start=0`. Wrap in `try/except → 416` and clamp bounds. *(Overlaps D5 — fixing the audio path under `ApiError` resolves both.)*
- **W2 — Raw `str(e)` to client** (`server.py:651,676,800,939,966,976`): SDK exceptions can carry URLs/response bodies/key fragments. Return a generic message, `logger.error(..., exc_info=True)` server-side. *(⚠️ plausibility of key leakage depends on SDK; the info-leak pattern is real regardless.)*
- **W3 — Lock held across `session.start()`** (`server.py:68-82`): RLock held through DB writes + WAV open + ScreenCaptureKit/sounddevice init (seconds), blocking `/api/status` polling and stop. Create session under lock, call `.start()` after release, re-check active state.
- **W4 — `executescript` implicit COMMIT** (`database.py:111-123`): commits any open tx; safe today, silently fragile if migration order changes. Wrap each step + its version bump in an explicit `with conn:`.
- **W5 — Body cap & missing Content-Length** (`server.py:537-553`): cap is skipped for chunked/no-length bodies. Currently safe (returns `b""`), but the documented control is inactive — add a 411 for `Transfer-Encoding: chunked` on mutating routes, or document the limitation.

## TEST GAPS (surfaced by the review)

- **TG1**: T1's `FakeSession.start()` sleeps 50ms inside the lock, not the real multi-second HW init — so it proves "one start wins" but cannot detect W3 (lock-held-across-IO).
- **TG2**: T3 tests handler methods directly, bypassing `do_*`/`_respond` — which is exactly why it missed B2. Add a test that drives the real dispatch (or a `BaseHTTPRequestHandler`-level test) and asserts status code on the wire.
- **TG3**: Unannotated `do_*` methods make mypy skip their bodies. Annotate them (`-> None`) so the type checker actually covers the routing layer; this alone would have flagged B2.
